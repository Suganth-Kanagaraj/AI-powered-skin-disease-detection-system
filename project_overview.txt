# Comprehensive Overview of the AI-Powered Skin Disease Detection System

## 1. Project Objective
The AI-Powered Skin Disease Detection system is an enterprise-grade, full-stack medical assistant designed to help detect, track, and report on skin diseases using artificial intelligence. It features a React-based frontend for end-users and medical staff, a FastAPI backend for high-performance API serving, integration with machine learning models for inference, and a comprehensive database setup for data persistence.

## 2. Overall Architecture
- **Frontend**: React.js configured with Vite. Uses Tailwind CSS for styling and Recharts for interactive dashboard analytics.
- **Backend**: FastAPI (Python) serving a RESTful API with async support.
- **AI Core**: Python scripts for AI model training and independent inference modules for disease detection.
- **Data Persistence**: Designed to work with both PostgreSQL (Relational Data) and MongoDB (NoSQL/Document data).
- **Deployment**: Fully containerized using Docker and orchestrated via Docker Compose.

## 3. Detailed Directory Breakdown

### `backend/`
The core API server built with FastAPI.
- **`app/main.py`**: The application entry point. It sets up the FastAPI application, CORS middleware, database connections on startup, and includes all the primary routers.
- **`app/api/`**: Contains the route controllers.
  - `auth.py`: User authentication, login, and registration endpoints.
  - `scan.py`: Endpoints for uploading images and processing disease scans.
  - `reports.py`: Endpoints to retrieve and manage medical reports.
  - `admin.py`: Administrative controls.
  - `dashboard.py`: Statistics and data points for frontend charts.
- **`app/core/`**: Core application settings, database configuration (`database.py`), and security logic.
- **`app/ai/`**: Contains the integration layer for the ML models, including `predictor.py` for inference and `preprocess.py` for image processing.
- **`app/schemas/` & `app/models/`**: Pydantic schemas for request/response validation and ORM models for database schema mapping.
- **`app/services/`**: The core business logic, keeping the API controllers thin.
- **`app/utils/`**: Helper utilities like `pdf_generator.py` for generating downloadable patient reports.
- **`requirements.txt`**: Backend Python dependencies.

### `frontend/`
The web user interface.
- **`src/main.jsx` & `App.jsx`**: Application bootstrap and top-level routing container.
- **`src/components/`**: Reusable UI elements, organized into logical sections (`dashboard/`, `admin/`, `layout/`, `report/`, `ui/`). Includes charts and structural elements like the Sidebar and Navbar.
- **`src/pages/`**: Main views representing specific routes (e.g., `Dashboard`, `ScanPage`, `ScanResult`, `Login`, `AdminPanel`).
- **`src/api/`**: Axios configuration (`axios.js`) and API service files that communicate directly with the FastAPI backend.
- **`src/routes/`**: Handles navigation and route protection (`ProtectedRoute.jsx`).
- **`package.json`**: Manages JavaScript dependencies (React, React-Router-DOM, Axios, Tailwind, Recharts).

### `ai-training/`
The machine learning sandbox.
- Contains scripts (`train.py`) and its own set of dependencies (`requirements.txt`) tailored to train the skin disease detection models independently of the main API server.

### `deployment/`
Infrastructure and hosting configuration.
- **`docker-compose.yml`**: Defines the multi-container application, spinning up the `backend` (port 8000), `frontend` (port 5173), `db` (Postgres 15), and `mongo` (MongoDB 6).
- **`nginx.conf`**: Configuration for Nginx, acting as a reverse proxy for staging or production deployment.

### `docs/`
- Contains project documentation such as `API_DOCUMENTATION.md`, outlining the primary API endpoints and workflows (e.g., Auth, Scan upload, History).

## 4. Workflows & Execution Path
1. **User Flow**: A user logs in through the React frontend (Auth handled by `backend/app/api/auth.py`). 
2. **Scanning**: The user navigates to the Scan page (`frontend/src/pages/ScanPage.jsx`) and uploads a skin image. 
3. **Processing**: The frontend posts this image to the `/api/scan` endpoint. The backend processes the image using `app/ai/preprocess.py` and feeds it into the model via `app/ai/predictor.py`.
4. **Reporting**: The result is saved to the database. The user can view the diagnosis via the dashboard or download a PDF report generated by `backend/app/utils/pdf_generator.py`.
